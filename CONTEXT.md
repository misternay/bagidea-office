# CONTEXT ‚Äî BagIdea Office Domain Glossary

> Glossary of the project's ubiquitous language. Domain terms only ‚Äî no
> implementation details. When a term below is used in code, docs, or discussion,
> it means *exactly* this.

## Meetings

### Meeting
A **live, interactive event**: a single in-progress run of the discussion engine
(`runDiscussion`). Has a lifecycle ‚Äî it *starts* (agents enter), *runs* (agents
take turns), and *ends* (adjourned). While a Meeting runs, the owner may speak
into it and may control it (pause / resume / end-early). A Meeting is ephemeral;
its durable residue is its **Meeting Log**.

Distinguish from a **Meeting Log** (the persisted transcript of a Meeting that
has ended) and from a **Sub-agent Log** (a bucket session in `@sub`, which is
never a live Meeting).

### Meeting Log
The **persisted artifact** a Meeting leaves behind: the transcript entry in the
group-session store plus the `workspace/meetings/g<key>.md` minutes file plus
its retrieval-index entry. A Meeting Log is **read-only** ‚Äî it is what you view
after a Meeting ends. A Meeting Log is *not* a Meeting: opening a log for
reading does not make it live.

### Sub-agent Log
A bucket session in `@sub` ‚Äî the transcript of a background ("ghost") sub-agent
job. Always read-only. Never a Meeting, never live. Shares the same pane
machinery as a Meeting Log but is a different kind of thing.

### Phase
A named **stage of a Meeting's structure** (e.g. `opening`, `discussion`,
`chat`, `user`, `user-response`). A Phase is a first-class, **persisted
property of each message** in a Meeting Log ‚Äî not a transient prompt concern.
The canonical set is fixed in one place; the markdown renderer and retrieval
treat all phases uniformly.

### Meeting Topic
The free-text subject of a Meeting. May be written by the owner, chosen from a
**Meeting Template**, or generated by an ambient/social tick.

### Meeting Template
A preset combination of **Meeting Topic + discussion depth** for a standard
meeting type (standup, retro, brainstorm, design-review, planning). Fills the
launcher; does not change the Phase structure.

## Tasks & Work

### Action Item
A unit of follow-up work **produced by a Meeting**, assigned to an **owner**
(an agent), with a text description and an open‚Üídone status. An Action Item is
**not scheduled** ‚Äî it fires when the owner acts, never by the clock. Distinct
from a **Work Order**, which is.

Canonical shape: `{ id, meeting, owner, text, status, created }` (deadline and
de-duplication are later concerns).

### Work Order (Job)
A **scheduled or recurring** unit of work with time semantics (`now` / `at` /
`every`). Fires by the clock. Distinct from an **Action Item**: a Work Order
answers *when*; an Action Item answers *who/what/by-whom*. Routing an Action
Item into the Work Order store is a category error ‚Äî they are different
entities.

### Mission Control
The **live-execution monitor**: a transient, in-memory board showing which
agents are currently running which work, colored by state. It is **not a task
store** ‚Äî it does not persist Action Items or Work Orders, and it is empty on
reload. "Creating a task in Mission Control" is not a real operation against a
store; it is a display event.

## Participation

### Owner (CEO)
The human user. In a Meeting, the Owner participates as a speaker with the id
`ceo`. The Owner's messages are a legitimate part of the Meeting Log, tagged
with the `user` phase.

### Run Id (transient "task")
A short correlation string minted per execution (e.g. `disc<ts>`, `t<n>`,
`soc<ts>`) used to group live activity for display. **Not a work item.**
Confusingly named `task` in places ‚Äî do not mistake it for an Action Item or
Work Order.
